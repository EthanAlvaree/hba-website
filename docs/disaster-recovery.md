# Disaster recovery runbook

_Last updated 2026-05-13._

This is the "the website / SIS just went sideways, what now" reference.
It is short on purpose: copy-paste-able, no philosophy.

## What we have

1. **Supabase automatic daily backups.** Supabase keeps point-in-time
   backups for the configured retention window (currently 7 days on the
   Pro plan, longer on higher tiers). These are the FIRST line of
   defense for most problems.
2. **Weekly full JSON snapshots** generated by the
   `/api/cron/db-backup` Vercel cron, written to the **`db-backups`**
   Supabase Storage bucket (private). Kept for ~12 weeks. These exist
   so we have an off-Supabase-recoverable copy of every row in every
   table.
3. **Storage buckets**: `application-documents`, `profile-photos`,
   `db-backups`. Supabase Storage also has automatic snapshots on Pro
   plan.

## Setup checklist (one-time, before first prod use)

- [ ] Create a private Storage bucket named **`db-backups`** in the
  Supabase dashboard. Do not make it public.
- [ ] Confirm `CRON_SECRET` is set in Vercel project env (already used
  by the M365 sync + parent digest crons).
- [ ] Confirm the daily Supabase backup is enabled (Pro plan or
  higher).
- [ ] After the first cron run, manually verify a snapshot file exists
  in `db-backups/`. It will be named `snapshot-<iso-timestamp>.json`.

## Common failure modes

### A. Bad UPDATE: someone mass-edited records they shouldn't have

**Recover from Supabase's point-in-time backup.** This is the fastest
and least lossy option.

1. Open the Supabase dashboard → Database → Backups.
2. Click "Restore" and pick a point in time before the bad UPDATE.
3. Supabase clones the database to that point. Note: this REPLACES the
   current state. Anything written between the restore point and now
   is lost (which is what you want here).

If the restore window doesn't reach back far enough, fall through to
the JSON snapshot path below.

### B. Table was DROPPED or schema was rolled back too far

Same as above: point-in-time restore from Supabase.

If point-in-time can't reach far enough back, the JSON snapshot:

1. Download the most recent acceptable `snapshot-*.json` from the
   `db-backups` bucket.
2. Identify which tables changed. Use `git log db/migrations/` to
   reconstruct any schema that may have been rolled back.
3. Re-apply migrations to a fresh Supabase project (or to a fresh
   schema in the existing project).
4. For each affected table, write a one-off SQL importer that takes
   the relevant `tables[X]` array from the snapshot JSON and inserts.

### C. Entire Supabase project is gone

This is the catastrophic case. It's why the JSON snapshots exist.

1. Provision a new Supabase project.
2. Apply every migration in `db/migrations/` in numeric order.
3. Pull the most recent snapshot from a colleague's local copy of the
   bucket (or restore the bucket itself if Storage survived).
4. Run a snapshot-import script (we don't have one in-repo yet — see
   "TODO" below) that takes the snapshot JSON and inserts each
   table's rows. Order matters because of foreign keys: profiles →
   students → parent_links → terms → courses → course_sections →
   enrollments → ...
5. Re-upload Storage objects (`application-documents`,
   `profile-photos`). These are NOT in the JSON snapshot — they're
   already in their own Supabase Storage buckets. If Storage is gone
   along with the project, applications and profile photos are lost;
   the school office can re-collect from email threads and yearbook
   photos as needed.
6. Point `HBA_SUPABASE_URL` and `HBA_SUPABASE_SERVICE_ROLE_KEY` in
   Vercel at the new project.
7. Redeploy.

### D. A specific row was deleted by accident

Two options:

- If Supabase point-in-time restore is available: do a temporary
  restore to a side project, copy the row out, paste back into prod.
- If not: open the most recent snapshot JSON, find the row by id, run
  an `INSERT` in the Supabase SQL editor.

## Verifying a snapshot is good

A snapshot you can't read is no snapshot at all. Do this once per
quarter:

1. Download the most recent `snapshot-*.json`.
2. Open in any JSON viewer. Confirm:
   - `generated_at` is recent.
   - `table_counts` shows non-zero for the high-value tables
     (profiles, students, enrollments, assignments, scores,
     attendance_records).
   - Spot-check one record in each.

If any of these are wrong, the cron is failing silently. Check Vercel
function logs for `/api/cron/db-backup`.

## Restoring from a JSON snapshot

`scripts/restore-from-snapshot.ts` does the heavy lifting. Walks the
snapshot in foreign-key-safe order, upserts every row by id, reports
per-table counts.

```sh
# Dry run first (no writes; just counts what would happen)
npx tsx scripts/restore-from-snapshot.ts ./snapshot-2026-05-13.json

# When the dry run looks right, commit:
HBA_SUPABASE_URL="https://<new-project>.supabase.co" \
HBA_SUPABASE_SERVICE_ROLE_KEY="<key>" \
  npx tsx scripts/restore-from-snapshot.ts \
    ./snapshot-2026-05-13.json --apply

# Restore a specific table only (e.g. the gradebook got corrupted):
npx tsx scripts/restore-from-snapshot.ts \
  ./snapshot-2026-05-13.json --apply \
  --only=assignments,scores,assignment_categories
```

The upsert is keyed on `id`, so re-running the script with the same
snapshot is safe — already-restored rows are overwritten in place
rather than duplicated. Safe to abort mid-restore and resume.

For point-in-time recovery (everything between the last snapshot and
the failure was lost), restore the snapshot first, then either:
- Replay any application/contact form submissions from the admin
  notification emails (search admissions@ inbox for the lost time
  window), or
- Accept the data loss and notify affected families.
